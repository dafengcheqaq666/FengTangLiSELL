# 项目开发重点要求

## 一、项目定位

本项目是“山野蜜境”鲜果商城，当前主推商品为“仁寿吴家祠蜂糖李”。项目由静态宣传页迁移为 Next.js 全栈应用，包含商品展示、鲜果篮、游客结算、订单查询、库存管理、微信/支付宝支付适配及单管理员后台。

所有改动应优先保证以下目标：

1. 保持现有页面视觉风格、中文文案和移动端响应式体验。
2. 保证订单金额、库存和支付状态准确，不能只实现前端演示效果。
3. 不降低支付回调、管理员登录和游客查单的安全性。
4. 新代码必须能通过类型检查、代码检查和相关测试。

## 二、技术栈与目录

- 框架：Next.js 16 App Router、React 19、TypeScript 严格模式。
- 数据库：PostgreSQL，使用 Prisma 6 管理模型和迁移。
- 校验：所有外部输入使用 Zod 校验。
- 测试：Vitest；数据库订单流程使用独立集成测试。
- 部署：Docker Compose，包括应用、PostgreSQL、迁移任务和过期订单定时任务。
- `app/`：页面、Route Handlers 和管理后台。
- `components/`：可复用 React 组件及客户端交互。
- `lib/`：订单、支付、鉴权、限流和数据库业务逻辑。
- `prisma/`：数据库模型、迁移和种子数据。
- `tests/`：单元测试与集成测试。

## 三、品牌与页面要求

- 面向顾客的产地品牌统一使用“仁寿吴家祠”，商品名使用“仁寿吴家祠蜂糖李”。
- 不得把配送省份列表中的行政区名称“贵州省”替换为品牌文字。
- 修改商品种子文案时，必须同步更新 `upsert` 的 `update` 与 `create` 数据，确保既有数据库和新数据库表现一致。
- 图片使用 `next/image`；新增图片放在 `public/images/`，必须提供准确的中文替代文本。
- 保持键盘可操作性、ARIA 属性和 `prefers-reduced-motion` 支持。
- 不应无理由重做现有布局、颜色、字体、动效或移动端断点。

## 四、订单与库存规则

- 金额统一使用人民币“分”的整数保存和计算，禁止使用浮点数作为数据库金额。
- 购物车价格只用于展示；创建订单时必须从数据库重新读取商品价格、库存和配送费。
- 创建订单必须在事务中原子预占库存，不能先查库存再无条件扣减。
- 待支付订单有效期为 15 分钟。释放库存前必须先关闭第三方支付单；关单失败时不得释放库存。
- 支付成功后同时扣减现有库存和预占库存，并保证重复回调幂等。
- 在线退款仅支持未发货订单整单原路退款；退款成功后恢复库存。
- 订单状态只能按照既定状态机流转，不得直接跳过支付或退款状态。

## 五、支付与安全要求

- 开发环境使用 `PAYMENT_MODE=mock`；生产环境禁止使用 Mock 支付。
- 微信支付按环境选择 JSAPI、H5 或 Native；支付宝按设备选择 WAP 或电脑网站支付。
- 浏览器返回页不能作为支付成功依据。只能接受验签后的异步通知或支付平台主动查询结果。
- 支付回调必须校验签名、应用/商户标识、订单号和支付金额，并支持重复通知。
- 私钥、证书、数据库连接、会话密钥和定时任务密钥只能通过环境变量提供，禁止提交到仓库。
- 游客订单详情必须校验订单访问令牌；重新查单需要订单号和收货手机号，并受限流保护。
- 管理员密码使用 Argon2 哈希；管理会话使用 Secure、HttpOnly、SameSite Cookie。
- 管理后台的商品、库存、配送、发货和退款变更必须写入审计日志。

## 六、数据库变更要求

- 修改 `prisma/schema.prisma` 后必须生成并提交新的迁移，禁止只运行 `prisma db push` 代替生产迁移。
- 迁移应向前兼容现有数据；涉及删除字段、表或数据时必须先明确迁移和回滚方案。
- 种子脚本必须可重复执行，不得重复创建同一商品、规格、配送区域或管理员。
- 禁止手工修改 Prisma 生成客户端和 `node_modules` 中的文件。

## 七、本地开发与运行

首次运行：

```powershell
Copy-Item .env.example .env
npm install
docker compose up -d postgres
npm run db:migrate
npm run db:seed
npm run dev
```

本地地址：

- 商城：`http://localhost:3000`
- 管理后台：`http://localhost:3000/admin`
- 健康检查：`http://localhost:3000/api/health`

Windows 环境必须保留 `next dev --webpack` 和 `next build --webpack`。默认 Turbopack 会因 Prisma 符号链接权限产生 `os error 1314`，不要随意移除 `--webpack`。

## 八、测试与交付标准

普通代码修改至少执行：

```powershell
npm run typecheck
npm run lint
npm test
```

涉及订单、库存、支付或数据库的改动，还必须在测试 PostgreSQL 上执行：

```powershell
npm run test:integration
```

准备交付或部署前执行：

```powershell
npm run build
docker compose config --quiet
```

交付前应确认：

- 首页、商品页、购物车、结算页和后台无运行时错误。
- 桌面端与移动端布局正常。
- 健康检查返回 HTTP 200。
- 数据库迁移和种子脚本可重复运行。
- 修改支付逻辑时已覆盖正常回调、重复回调、金额不匹配、超时关单和退款失败场景。

## 九、改动边界

- 不提交 `.env`、数据库备份、日志、`.next/`、npm 缓存或其他构建产物。
- 不绕过现有服务层直接从客户端修改订单、库存或支付状态。
- 不在缺少商户证书时伪造真实支付或退款成功。
- 不删除现有功能、数据模型或安全校验，除非任务明确要求且已补充相应迁移和测试。
- 保持改动聚焦，避免与任务无关的大规模重构或依赖升级。
